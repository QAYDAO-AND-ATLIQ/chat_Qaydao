## 3. المعمارية المقترحة (3 خيارات + التوصية)

### الخيار 🅰: Chatwoot Captain (المدمج)
**الكيفية:** Chatwoot 2025+ عنده AI agent مدمج اسمه "Captain"، يحتاج فقط OpenAI API key.

| المميزات | العيوب |
|----------|--------|
| ✅ نشر في 5 دقائق | ❌ يحتاج خطة Cloud مدفوعة (~$30/مستخدم/شهر) |
| ✅ مدمج بالكامل | ❌ Self-hosted version لازال beta للـ Captain |
| ✅ يستخدم Help Center تلقائياً | ❌ تخصيص محدود |
| | ❌ لا يدمج مع نظام QAYDAO الداخلي (المنتجات، الطلبات) |

**التقييم:** غير مناسب — لأن QAYDAO self-hosted، Captain غير مستقر بعد على هذه النسخة.

### الخيار 🅱: n8n + OpenAI + Chatwoot Webhooks (المرن)
**الكيفية:**
```
Customer Message → Chatwoot Webhook → n8n Workflow
  → check business hours
  → if outside hours:
    → fetch conversation history (Chatwoot API)
    → fetch product/order context (QAYDAO DB via API)
    → call OpenAI with system prompt + RAG
    → post reply back to Chatwoot
    → tag conversation "ai_handled"
    → if escalation needed → assign to agent + private note
```

| المميزات | العيوب |
|----------|--------|
| ✅ n8n موجود بالفعل (`n8n.qaydao.com`) | ⚠️ يحتاج build + maintain workflow |
| ✅ تحكم كامل في prompts/logic | ⚠️ logs مفصلة لكل قرار |
| ✅ تكامل سهل مع باقي أنظمة QAYDAO | |
| ✅ Self-hosted بالكامل | |
| ✅ PDPL-friendly | |

**التقييم:** ⭐ **التوصية** — يستفيد من البنية الحالية ويعطي تحكم كامل.

### الخيار 🅲: Custom Python Microservice
**الكيفية:** FastAPI service يستقبل webhooks من Chatwoot ويرد عبر OpenAI.

**التقييم:** قوي لكن overkill — n8n يعطي 90% من الفائدة بـ 30% من الجهد. ابقَ مع 🅱.

## 4. مصادر تغذية المعلومات (Knowledge Base Strategy)

**القاعدة الذهبية:** "AI is only as good as its data — Garbage in, Garbage out"

### 4.1 المصادر المقترحة (مُرتَّبة بالأولوية)

#### المصدر 1: قاعدة بيانات المنتجات (الأهم)
- **المصدر:** Salla API أو qaydao.com/products
- **المحتوى:** اسم المنتج، الأبعاد، الألوان، السعر، مدة التصنيع، ETA الحاوية
- **التحديث:** كل 6 ساعات (تلقائي)
- **استخدام AI:** "كم سعر كرسي Q_PLUS؟" → بحث مباشر

#### المصدر 2: حالة الطلب الحالي (Critical للعميل)
- **المصدر:** نظام الطلبات الداخلي + tracking
- **المحتوى:** رقم الطلب، مرحلة التصنيع، ETA الشحن، رقم الحاوية
- **التحديث:** real-time عبر API
- **استخدام AI:** "وين طلبي #250000000؟" → "طلبك في منتصف التصنيع، يصل خلال 35 يوم"

#### المصدر 3: FAQ + سياسات الشركة
**يجب بناؤها** — لا توجد help center حالياً منظم. المحتوى المقترح:
- ⏱️ **مدة التصنيع والشحن:** "30 يوم تصنيع + 45-60 يوم شحن بحري"
- 💰 **سياسة الاسترجاع/الإلغاء:** الشروط والمدة
- 🛒 **طرق الدفع:** Mada, Apple Pay, STC Pay, Tabby/Tamara
- 📦 **الشحن:** متى تخرج الحاوية، مناطق التوصيل، أسعار الشحن
- 🛠️ **التركيب:** هل يأتي جاهز؟ هل يوجد فني؟
- 🏬 **الفروع:** أين توجد، ساعات العمل
- 📞 **التواصل:** support@, ساعات الرد، طرق التصعيد

**ملاحظة من QA Audit:** هذه الأسئلة بالضبط هي الأكثر تكراراً في Ghost Closures:
- "كم مدة التوصيل؟" #464
- "هل لكم فروع بالرياض؟" #372
- "هل القطع تجي جاهزه او تحتاج تركيب؟" #1649
- "كراسي طبية لعيادات لون بيج 5 كراسي" #1610

#### المصدر 4: المحادثات التاريخية (Best Practice Mining)
- **المصدر:** Chatwoot DB (33,950 رسالة تاريخية)
- **الاستخدام:** استخراج أفضل ردود الـ agents (خاصة قوالب مروة) كـ few-shot examples
- **التحديث:** أسبوعي

### 4.2 آلية التغذية: RAG vs Fine-tuning vs Function Calling

| الأسلوب | متى يستخدم؟ | في QAYDAO |
|---------|--------------|------------|
| **RAG (Retrieval-Augmented Generation)** | معلومات تتغير كثيراً | ✅ **الأنسب** للمنتجات + FAQ |
| **Function Calling** | استعلامات live من DB | ✅ للطلبات + tracking |
| **Fine-tuning** | تخصيص lhgة/أسلوب | ⏸️ لاحقاً (بعد 3 أشهر) |
| **System Prompt** | شخصية AI + قواعد | ✅ مطلوب فوراً |

**الاستراتيجية المقترحة:** RAG (للمعرفة الثابتة) + Function Calling (للبيانات الحية) + System Prompt قوي.

## 6. الامتثال للأنظمة (PDPL Compliance)

### 6.1 المتطلبات الإلزامية (PDPL السعودي + Meta WhatsApp Policy 2026)

| المتطلب | كيف يُحقَّق في QAYDAO AI |
|---------|---------------------------|
| ✅ **Opt-in واضح** | رسالة افتتاحية: "هذا رد آلي من QAYDAO AI، إجابتك بأي شيء = موافقة" |
| ✅ **شفافية AI** | كل رسالة من AI تبدأ بـ "🤖 QAYDAO AI:" — لا تتظاهر بأنها بشر |
| ✅ **Human Handoff** | زر/كلمة "تكلم مع موظف" يصعّد فوراً |
| ✅ **Data Localization** | البيانات تبقى في DB السعودي/Hostinger، فقط النص يُرسل لـ OpenAI |
| ✅ **Logs كل قرار** | تسجيل كل response في DB مع timestamp + tokens used |
| ✅ **No Bank Data** | AI **لا يطلب أبداً** IBAN/بطاقة — يوجّه لنموذج آمن |
| ✅ **Task-Specific** | AI **لا** يردّ على أي شيء خارج: منتجات، طلبات، شحن، سياسات — كل شيء آخر يُحوَّل |
| ✅ **Right to Erasure** | عند طلب العميل، حذف رسائله من الـ training/logs |

### 6.2 ⚠️ تحذير حرج: Meta WhatsApp Policy (يناير 2026)

> منذ 15 يناير 2026، Meta منعت "general-purpose AI assistants" على WhatsApp Business API.
> **مسموح فقط:** task-specific bots (تتبع طلب، استفسار منتج، booking، إلخ).

**التأثير على QAYDAO:**
- ✅ نمط الاستخدام المقترح **متوافق** (task-specific)
- ❌ لا يجب أن يردّ AI على: "ما رأيك في الحياة؟" أو محادثات عامة
- ✅ يجب أن يبقى scope محدد: منتجات + طلبات + سياسات

## 7. منطق التحويل (Escalation Logic)

AI **يحوّل فوراً للبشر** في الحالات التالية:

| Trigger | المثال | الإجراء |
|---------|--------|----------|
| 🔴 **غضب/شكوى** | "زعلانه منكم"، "بشتكي" | تحويل فوري + label "عاجل" + تنبيه مشرف |
| 🔴 **قانوني** | "محكمة"، "تجاري"، "نظام التجارة" | تحويل فوري للمشرف |
| 🔴 **استرجاع/إلغاء** | "ارجع المبلغ"، "كنسل الطلب" | تحويل + جمع رقم الطلب أولاً |
| 🔴 **VIP** | عميل في customAttributes "is_vip=true" | تحويل فوري لمروة |
| 🔴 **بنك/IBAN** | كشف نمط `SA[0-9]{20}` | توجيه لنموذج آمن + تحويل |
| 🟡 **AI غير واثق** | confidence < 70% في 3 ردود متتالية | تحويل بأدب |
| 🟡 **طلب صريح** | "أبغى أكلم موظف" | تحويل + رسالة "نحوّلك للفريق" |
| 🟡 **5 رسائل متبادلة** | لو AI ما حل القضية | تحويل تلقائي مع summary |
| ⏰ **دخول الدوام** | 9:00 صباحاً | تحويل لكل المحادثات المفتوحة |
